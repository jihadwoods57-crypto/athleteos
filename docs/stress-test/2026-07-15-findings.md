## Verdict: **GO WITH FIXES**

OnStandard is a **mature, security-aware codebase**. Every load-bearing boundary I could inspect
statically or probe read-only held up: server-validated auth on every edge function, `service_role`
absent from the client, consistent HTML-escaping, Stripe signature verification, a PII-firewalled
telemetry endpoint, and a deliberate migration-by-migration hardening history. Prod DB state matches
the repo (only the newest migration 0063 pending). No P0 was found in the surfaces examined.

Two things gate an unconditional GO, and both are **operational/config**, not code defects:

- **B1 (P1, verify immediately):** the deployed `stripe-webhook` returns **HTTP 500** to an
  unsigned request, which means its Stripe secrets are not set on the prod function. If billing is
  live, paid customers won't get entitlements. If billing isn't launched, the endpoint should ack
  instead of 500. **Founder must confirm which.**
- **Ring-1 not yet run:** load ceiling, concurrency error-rates, cross-role write probes, and the
  soak invariants are unproven until the suite runs against a disposable project (one founder
  action to provision; then it's automated).

## Findings (severity-ranked)

### B1 ‚Äî [P1] `stripe-webhook` returns 500 on prod (billing secrets likely unset)
**Evidence:** `sim/stress/edge-failclosed.mjs` against prod ‚Äî 9/10 functions reject cleanly; `stripe-webhook` POST with no signature ‚Üí **HTTP 500** (expected 400 "missing signature").
**Root cause:** `supabase/functions/stripe-webhook/index.ts:144-147` checks config **before** the
signature and returns 500 if `STRIPE_SECRET_KEY`/`STRIPE_WEBHOOK_SECRET` is missing (URL + service
role are auto-injected, so the missing one is a Stripe secret). The module-level `new Stripe('')`
(line 46) would also throw on boot if the key is empty.
**Failure scenario:** if billing is live, Stripe calls this on `checkout.session.completed`, gets a
500, retries with backoff, and eventually gives up ‚Üí the customer's `subscriptions` row is never
written ‚Üí **they paid and got no access**. A 500 also triggers Stripe retry storms.
**Fix:** confirm whether billing is launched. If yes: `supabase secrets set STRIPE_SECRET_KEY=‚Ä¶
STRIPE_WEBHOOK_SECRET=‚Ä¶` and re-probe. If no: return 200/503 for the unconfigured case so Stripe
doesn't retry (change the `!STRIPE_WEBHOOK_SECRET` branch from 500 to a 200 ack). **Do not** fix
prod secrets without founder sign-off.

### S1 ‚Äî [P2] Live 3rd-party secrets co-mingled in the Expo `.env`
**Evidence (auth sub-audit):** `.env:11-15` holds real OpenAI (`sk-proj-‚Ä¶`), Cloudflare (`cfat_‚Ä¶`),
and Higgsfield keys in plaintext. Verified **not** `EXPO_PUBLIC_*`, **not** referenced in `src/`, and
`.env` is gitignored + never committed ‚Üí **no client-bundle leak today**.
**Risk:** blast radius. One accidental `EXPO_PUBLIC_` prefix, a stray client import, or a `.gitignore`
regression turns these into a live leak; and they sit on disk in plaintext regardless. The project's
own `.env.example` says server secrets belong in `supabase secrets`.
**Fix:** move OpenAI/Cloudflare/Higgsfield to a separate untracked env used by `web/landing-src/hf-*.mjs`;
rotate all three as hygiene; keep only `EXPO_PUBLIC_*` in the app `.env`.

### S2 ‚Äî [P2] Unencrypted PII / health-adjacent data in WebView localStorage
**Evidence:** `proto/redesign-2026-07/js/state.js` (`RT`: userId, email, authRole, identity) and
`js/day.js` (`onstd-day-*` cache: meals, scores, weight) persist to **localStorage** (unencrypted on
disk). Session **tokens** are correctly in the OS Keychain/Keystore ‚Äî this is about the data cache,
not tokens.
**Risk:** email + cached nutrition/score/weight readable from a device backup, lost/jailbroken device,
or forensic extraction.
**Fix:** route the sensitive cache through the SecureStore bridge (it already allows `onstd-*` keys),
or minimize what's persisted (drop `email`, cap cached day data).

### S3 ‚Äî [P2] No environment separation; prod URL+anon key is the committed shipping fallback
**Evidence:** `proto/redesign-2026-07/index.html:19-22` hardcodes the prod Supabase URL + anon key as
the `window.__SUPABASE` fallback; `eas.json` never sets `EXPO_PUBLIC_SUPABASE_URL/ANON_KEY` for any
profile, so **dev/preview/production all hit the one prod project**. Anon key is RLS-gated ‚Üí not a
secret leak, but a config hazard (rotating the anon key means editing committed source; no safe place
to load-test the real app).
**Fix:** source URL/anon from env in all EAS profiles; stand up a standing non-prod project; treat the
`index.html` literal as a localhost-only fallback. (This is also what makes Ring-1 testing require a
throwaway project each time.)

### R1 ‚Äî [P3] Stale-auth: render gate trusts persisted `userId`, not a live session
**Evidence:** `js/router.js:82` gates renders on `RT.userId` from localStorage. Boot **does** reconcile
via `getSession()` and `onAuthStateChange('SIGNED_OUT')` wipes state ‚Äî so most of the gap is closed.
Residual: a server-side revocation (password changed elsewhere, admin revoke) isn't reflected until the
next refresh/data call; the no-session boot branch routes to Welcome without clearing `RT.userId`. RLS
still blocks all server data throughout, so exposure is limited to locally-cached data in a half-authed
shell.
**Fix:** clear `RT.userId`/run `_wipeUserScopedState` on the no-session boot branch; optionally confirm
session freshness before painting an authed shell.

### R2 ‚Äî [P3] WebView CORS-bypass flags amplify any DOM-XSS to exfiltration
**Evidence:** `src/proto/ProtoApp.tsx:213-221` sets `originWhitelist=['*']` + file-access flags (needed
for the `file://` ES-module proto). Any script in the WebView can reach any host, so a DOM-XSS becomes
data/session exfil, not just defacement. **Well-mitigated today** by the consistent `esc()` escaper
(`components.js:10`, verified across coach/meal/roster/chat sinks) and the bridge key allowlist.
**Fix:** keep the `esc()` discipline; add a CI lint flagging unescaped `${‚Ä¶}` inside `innerHTML` template
strings; tighten `originWhitelist` toward the file origin if the loader tolerates it.

### R3 ‚Äî [P3] Console bridge forwards all WebView logs to native; R4 biolock fails open
**Evidence:** `ProtoApp.tsx:18-26` posts every WebView `console.*` to native (latent PII channel in
release builds; nothing sensitive logged today). `ProtoApp.tsx:56-64` unlocks on a SecureStore read
fault (fail-open Face ID gate over an already-encrypted session).
**Fix:** gate the console forwarder to `__DEV__`; consider fail-closed biolock with a retry.

### O1 ‚Äî [P3] `analytics-ingest` per-IP rate limit is per-isolate (best-effort)
**Evidence:** `analytics-ingest/index.ts:57-64` keeps the rate-limit `Map` in isolate memory; isolates
recycle, and the `hits` map isn't evicted within an isolate's life. The batch cap + PII firewall +
event whitelist are the real controls, so abuse is bounded to whitelisted, structurally-PII-free rows.
**Fix:** acceptable as-is; if analytics spam becomes real, move the limiter to a durable store.

## Verified strengths (do not regress)

- **Every edge function authorizes via server-validated `supabase.auth.getUser(token)`** ‚Äî a forged
  JWT `sub` cannot impersonate or buy AI calls; client-supplied `user_id` is never trusted.
- **`service_role` appears nowhere in the repo or client bundle**; only the RLS-gated anon key ships.
- **`stripe-webhook` verifies the Stripe signature** against the raw body (`constructEventAsync`),
  `service_role` is the only writer of `subscriptions`, referral rewards are idempotent, DB errors
  return 500 so Stripe retries, no detail leaks. (The B1 issue is deployment config, not this code.)
- **9/10 edge functions fail closed** (clean 4xx on unauth/malformed) ‚Äî proven at runtime on prod.
- **Consistent HTML-escaping** (`esc()`) at every user-data `innerHTML` sink, incl. coach messages
  and meal-thread bodies; `camera.js` deliberately uses `textContent`.
- **`guardian-verify` (COPPA) is textbook** ‚Äî 128-bit opaque token, two-step GET/POST, regex+escape,
  rate-limited, idempotent; a minor cannot self-verify (`0035` closed the token self-read hole).
- **Migration hardening arc is real and applied:** 0013 closed `orgs_read using(true)` enumeration,
  0035 closed the notify()-forgery and function-EXECUTE holes and made future functions
  secure-by-default. Prod has **0001‚Äì0062 applied**; only 0063 pending.
- **Session tokens in the OS Keychain** (chunked), refresh + rotation configured, boot fetches role
  server-side before routing, `verify_jwt=false` only where correct and **pinned in config.toml**.
- **Prod read latency healthy:** days p50 83ms/p95 96ms; profiles p50 198ms (p95 713ms is
  cold-connection noise on n=12); edge preflight ~220ms p50.
- **Ring-0 fully green:** typecheck clean, **1833 tests / 150 suites pass**, iOS bundle exports.
